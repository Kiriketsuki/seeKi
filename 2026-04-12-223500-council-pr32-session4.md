## Live Findings

### Finding S4-1 — CD-1 Bug 1: orphaned-backup+evolved-config regression (UNANIMOUS)

**Scenario that breaks:**
1. Run #1 setup: CONFIG_BACKUP absent → copies config A (user's current config) → backup; writes test config → CONFIG_DST. CITE: `frontend/e2e/global-setup.ts` L:103-106
2. Run #1 teardown crashes (SIGKILL, OOM, power loss) before `renameSync(BACKUP→DST)` executes. CITE: `frontend/e2e/global-teardown.ts` L:43
3. User manually repairs `seeki.toml` to config B (e.g. different DB host).
4. Run #2 setup: `CONFIG_BACKUP` = A, `CONFIG_DST` = B. `A ≠ B` → `shouldBackup = !true || (A===B) = false` → backup preserved as stale A. CITE: `frontend/e2e/global-setup.ts` L:103-104
5. Setup writes test config → CONFIG_DST.
6. Run #2 teardown: `renameSync(BACKUP=A, CONFIG_DST)` — config A overwrites CONFIG_DST, config B permanently lost. CITE: `frontend/e2e/global-teardown.ts` L:43

Same path triggers when a developer has a pre-existing `seeki.toml.user-backup` that differs from the current `seeki.toml`.

**Root cause**: The guard compares `BACKUP` content against `DST` content. It should compare `DST` against `configContent` (the test config about to be written — already in scope at L86): if `DST ≠ configContent`, DST is user config and must be captured; if `DST === configContent`, DST is the leaked test config from an interrupted run — preserve backup.

**Severity**: MEDIUM data loss — user's edited config silently destroyed with no warning.

## Arbiter Adjudication

### On CD-1 Bug 1

The orphaned-backup+evolved-config scenario is a genuine regression introduced by d72decb. The original pre-d72decb code (`fs.copyFileSync(CONFIG_DST, CONFIG_BACKUP)` unconditional) correctly handled this case — it would have captured config B at step 4. The d72decb guard, while correctly addressing the original CD-1 Critical Discovery, regresses this path. All parties agree. **This is a merge blocker requiring a code fix.**

The correct fix is to compare `CONFIG_DST` content against `configContent` (the expanded test config computed at L86-95, already in scope): if `DST ≠ configContent`, DST is user config and should be backed up; if `DST === configContent`, the prior interrupted run left test config in DST — preserve the existing backup.

### On O-1 Severity

The facts are agreed. The severity dispute turns on one question: does introducing a new destructive operation into the primary invocation path create a documentation obligation that belongs in the same commit?

**Ruling**: Yes. The Justfile `test-e2e` recipe is the documented entry point for running E2E tests. Before d72decb, running `just test-e2e` did not modify DB data — it issued reads only. d72decb adds a `TRUNCATE + INSERT` to every `just test-e2e` execution. A developer who configures `.env.test` reasonably assumes the test suite is non-destructive (it was, until now). The `[global-setup] Applying seed.sql...` log communicates an action but not its consequence. This is a documentation obligation introduced by d72decb and belongs in the same PR — either as a Justfile comment or a more explicit log message. **Condition, addressable in one line.**

## Conditions

**Condition 1 (REQUIRED — code fix): CD-1 Bug 1 — fix the backup guard**

Replace the `backup === dst` comparison with `dst !== configContent` in `global-setup.ts`.

CITE: `frontend/e2e/global-setup.ts` L:98-111

**Before** (current d72decb code):
```typescript
  if (fs.existsSync(CONFIG_DST)) {
    const shouldBackup = !fs.existsSync(CONFIG_BACKUP) ||
      fs.readFileSync(CONFIG_BACKUP, 'utf-8') === fs.readFileSync(CONFIG_DST, 'utf-8');
    if (shouldBackup) {
      fs.copyFileSync(CONFIG_DST, CONFIG_BACKUP);
      console.log(`[global-setup] Backed up existing ${CONFIG_DST} → ${CONFIG_BACKUP}`);
    } else {
      console.log(`[global-setup] Preserving existing ${CONFIG_BACKUP} (differs from current ${CONFIG_DST}; likely user config from a prior interrupted run)`);
    }
  }
```

**After** (corrected):
```typescript
  // CD-1 safety: back up DST before overwriting with test config.
  // Compare DST against the test config we are about to write (configContent):
  //   - DST !== configContent → DST is user's real config (original or edited since
  //     last run) — capture it as backup.
  //   - DST === configContent → a prior interrupted run left test config in DST;
  //     the existing backup already has the real user config — preserve it.
  if (fs.existsSync(CONFIG_DST)) {
    const dstContent = fs.readFileSync(CONFIG_DST, 'utf-8');
    if (!fs.existsSync(CONFIG_BACKUP) || dstContent !== configContent) {
      fs.copyFileSync(CONFIG_DST, CONFIG_BACKUP);
      console.log(`[global-setup] Backed up existing ${CONFIG_DST} → ${CONFIG_BACKUP}`);
    } else {
      console.log(`[global-setup] Preserving existing ${CONFIG_BACKUP} (DST already contains test config from prior interrupted run)`);
    }
  }
```

Note: `configContent` is already in scope at this point (computed at L86–95). No new reads required.

**Scenario verification:**

| Scenario | `!existsBackup` | `dst !== configContent` | `shouldBackup` | Outcome |
|---|---|---|---|---|
| First run (no backup, DST=user config) | true | true | true | Backs up user config ✓ |
| Interrupted run (backup=user, DST=test_config=configContent) | false | false | false | Preserves user backup ✓ |
| Normal 2nd run (backup=user, DST=user after teardown) | false | true | true | Re-copies user config (no-op) ✓ |
| User edits between runs (backup=old, DST=new) | false | true | true | Backs up new config ✓ FIXED |
| Orphaned backup + user edits (backup=A, DST=B≠configContent) | false | true | true | Backs up B ✓ FIXED |
